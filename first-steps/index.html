Introducción

Git

Git es un software de control de versiones distribuido pensado para la eficiencia y confiabilidad del mantenimiento de versiones de aplicaciones cuando éstas tienen un gran número de archivos de código fuente.

El diseño de Git resulta de la experiencia del diseñador de Linux, Linus Torvalds, manteniendo una enorme cantidad de código distribuido y gestionado por mucha gente, que incide en numerosos detalles de rendimiento.

Características

  • Soporte al desarrollo no lineal. Las ramas en Git van muy baratas (crear una rama y fusionar posteriormente es trivial).
  • Desarrollo distribuido. Cada desarrollador tiene una copia local del repositorio.
  • Compatibilidad con protocolos y sistemas ya existentes: HTTP, FTP, rsync, o SSH.
  • Autenticación criptográfica del historial. Una vez hecho un commit no es posible hacer cambios sin ser detectado.

Plataformas

¿En qué plataformas funciona Git?

  • Linux. Es su platafoma nativa.
  • Sistemas POSIX. Con ello cualquier variante Unix: BSD, Solaris, Darwin (MacOSX), etc.
  • Windows. A través de alguna de estas implementaciones: msysGit, TortoiseGit, encima de Cygwin, o JGit.

Instalación

La instalación de Git dependerá de la plataforma, pero suele ser un proceso trivial ya que viene empaquetado en todas las distribuciones.

# En Ubuntu/debian
$ sudo apt-get install git

# En Archlinux
$ sudo pacman -S git

En Windows o MacOSX puede ser un poco más laborioso, pero no mucho más.

Comandos básicos

Inicializar un repositorio

git init. Es el comando que nos permite iniciar el control de versiones sobre un directorio. 

# Creamos un directorio y un archivo
$ mkdir helloworld
$ cd hellworld
[~/helloworld]$ echo "Hello World" > README

# Inicializamos el repositorio
[~/helloworld]$ git init
Initialized empty Git repository in /home/dave/helloworld/.git/

Clonar un repositorio

glit clone. Este comando obtiene una copia local de un repositorio remoto. 

# Clonamos el repositorio fuente de esta presentacion
$ git clone git@github.com:tombatossals/git-in-practice.git
Cloning into 'git-in-practice'...

$ cd git-in-practice/
[~/git-in-practice]$ ls

El directorio .git

Una de las cosas que más llaman la atención es lo limpio que queda todo en tu directorio de trabajo. Si al clonar un repositorio lo estamos obteniendo de manera completa...

  • ¿cómo es que lo ha obtenido tan rápido?
  • ¿dónde está la base de datos de revisiones?

Ejercicio: Ejecuta un "find .git" en tu directorio de trabajo y echa un vistazo a la estructura de archivos que hay creada.

Los "snapshots"

IMPORTANTE. Aquí tenemos una diferencia fundamental con respecto al modo de trabajar de otros SMC. Git trabaja sacando instantáneas de todo el árbol de archivos modificados entre commits. Otros sistemas de control de revisiones trabajan a nivel de archivo individual.

  • Es más costoso examinar el historial de cambios de un archivo individual que de todo el proyecto.

El índice

git status

El índice nos informa sobre las diferencias actuales entre el repositorio y tu directorio de trabajo.

Lo utilizaremos para construir un commit que agrupe modificaciones en varios archivos.

Cuando hagamos el commit, se enviará lo que haya en el índice, no lo que haya en nuestro directorio de trabajo.

El comando que nos muestra la información sobre el índice es git status.

Trabajando con Git

git config

Vamos a establecer algunas variables básicas de trabajo con las directivas de configuración global. 

$ git config --global user.name "Peter Pan"
$ git config --global user.email "peter.pan@nuncajamas.com"
$ git config --global color.ui true

# Se modifica el archivo de configuración global de git
$ cat $HOME/.gitconfig 
[user]
    name = Peter Pan
    email = peter.pan@nuncajamas.com
[color]
    ui = true
